Case Study 01 — Lab Demonstration

Mapping the Invisible: AD Attack Path Analysis

Summary

Most Active Directory compromises exploit configuration, not software vulnerabilities. This case study walks through using attack path mapping to make invisible privilege relationships visible: collecting a graph of a representative domain, querying it for the misconfigurations attackers rely on, and turning flat object lists into a prioritized map of who can reach Tier 0 and how.

Toolchain

ToolRole
BloodHound CE + Neo4jGraph database and analysis UI
SharpHoundOn-host Windows collector
bloodhound-pythonRemote Linux-based collection — no binary dropped
PingCastleDomain risk scoring for executive reporting
Custom CypherTargeted hunting beyond built-in queries

Methodology

Collection was run remotely from a Linux host using bloodhound-python with -c All to capture ACLs and session data. Results were ingested into Neo4j and analyzed in three passes: built-in queries, custom Cypher, and shortest-path analysis to quantify exposure. Session data was collected during business hours to capture active privileged sessions.

Findings

Finding 01
ACL-based privilege escalation chains
The HelpDesk group held GenericAll over a Server Admins member, which held WriteDacl on Domain Admins. Invisible across three separate admin screens; one shortest-path query made the chain explicit.
MATCH p=shortestPath((n)-[*1..]->(g:Group)) WHERE g.objectid ENDS WITH '-512' RETURN p
Finding 02
Shadow admins via dangerous ACL edges
A former contractor account held ForceChangePassword over a privileged service account. No group membership, no visible privilege in the org chart — only visible in the graph.
Finding 03
DCSync rights on non-DC principals
An Azure AD Connect sync account carried broader replication rights than required, plus a secondary orphaned account from a failed troubleshooting session retained GetChanges and GetChangesAll.
MATCH p=(n)-[:DCSync|GetChanges|GetChangesAll]->(d:Domain) WHERE NOT n.objectid ENDS WITH '-516' RETURN n.name, labels(n)
Finding 04
Kerberoastable accounts on privileged paths
Three service accounts with SPNs were cross-referenced against their outbound paths. Two sat on a path to Domain Admins, elevating them from low-priority noise to critical findings.
Finding 05
Unconstrained delegation on non-DC host
A legacy application server retained unconstrained delegation from a migration five years prior. The application was gone; the flag was not.
Finding 06
Privileged sessions on non-Tier-0 workstations
Domain Admins were logging into standard helpdesk workstations. Session data revealed active credential exposure on machines not managed to Tier 0 standards.

Outcome

A meaningful percentage of ordinary domain users held a path to Domain Admin through permission chains no administrator had ever intentionally created. Every path was a configuration decision — not a patch-level vulnerability — which means every one is fixable without purchasing additional tooling.

Case Study 02 — Lab Demonstration

Closing the Paths: AD Security Remediation

Summary

Attack path mapping shows where an adversary can go. This case study covers the full engineering loop: root-cause analysis for each misconfiguration class, targeted remediation with specific tooling, and post-remediation verification via re-collection that confirms the paths are genuinely closed — not just changed.

Principle

Remediation is a prioritization exercise driven by path criticality, not a flat checklist. Shortest, highest-value paths to Tier 0 are closed first. Every change is tested for functional impact before and after, and re-collection confirms the structural closure. Over-remediation without testing creates rollback pressure — the enemy of permanent fixes.

Remediation by Finding Class

Finding 01 — ACL chains
Restore inheritance; remove non-default ACEs
Non-inherited ACEs were documented via AD ACL Scanner, then removed using PowerShell’s AD: drive. Where inheritance was broken to enable the over-permissioned ACE, restoring inheritance propagated cleanup automatically.
$acl = Get-Acl "AD:CN=Server Admins,OU=Groups,DC=lab,DC=corp,DC=local" $ace = $acl.Access | Where-Object { $_.IdentityReference -eq 'LAB\HelpDesk' -and $_.ActiveDirectoryRights -eq 'GenericAll' } $acl.RemoveAccessRule($ace) Set-Acl -Path "AD:CN=Server Admins,..." -AclObject $acl
Finding 02 — Shadow admins
Strip memberships from stale accounts; remove ACEs
Disabled accounts with dangerous outbound edges are never operationally justified. Group memberships and ACEs were stripped; accounts flagged for deletion through change control.
Finding 03 — DCSync rights
Revoke replication rights from non-DC principals
Used dsacls.exe to remove GetChanges / GetChangesAll from the orphaned troubleshooting account entirely. The primary sync account was scoped to minimum required rights based on the actual sync configuration in use.
dsacls "DC=lab,DC=corp,DC=local" /R "LAB\stale_sync_account"
Finding 04 — Kerberoastable accounts
Migrate to gMSA; rotate remaining service account credentials
Group Managed Service Accounts rotate their own 240-character passwords automatically, eliminating the offline-cracking surface entirely. For accounts that could not be migrated immediately, credentials were rotated to a 30+ character random value and SPNs were scoped to minimum required.
New-ADServiceAccount -Name "gMSA_AppService" ` -DNSHostName "appservice.lab.corp.local" ` -PrincipalsAllowedToRetrieveManagedPassword "AppServers_Hosts" Set-ADUser -Identity "svc_appservice" ` -ServicePrincipalNames @{Remove="HTTP/appservice.lab.corp.local"}
Finding 05 — Unconstrained delegation
Remove or convert to constrained / RBCD
Unconstrained delegation removed from the legacy host. All Tier 0 accounts were added to the Protected Users group and flagged as “sensitive and cannot be delegated.”
Set-ADComputer -Identity "LEGACYAPP01" -TrustedForDelegation $false Set-ADAccountControl -Identity "DA_account" -AccountNotDelegated $true Add-ADGroupMember -Identity "Protected Users" -Members "DA_account"
Finding 06 — Privileged sessions
Tier 0 isolation via GPO and PAW model
A Group Policy Object applied “Deny log on locally” and “Deny access from the network” for Domain Admins to all OUs except the Tier 0 PAW OU. Domain Admins may initiate interactive sessions only from the dedicated, hardened workstation. Credential Guard enabled across all domain-joined endpoints as an interim control.

Verification

Post-remediation SharpHound collection was run immediately after changes and again 48 hours later to catch any permission re-propagation via AdminSDHolder. The user-path-to-DA Cypher query returned zero for direct paths. PingCastle score was re-run to provide a trending metric for client reporting.

Sustainability

For multi-tenant MSP environments: scheduled monthly re-collection, a templated hardening GPO baseline applied to all newly onboarded tenants, and quarterly AdminSDHolder audits to prevent the single-misconfiguration-that-multiplies failure mode.
